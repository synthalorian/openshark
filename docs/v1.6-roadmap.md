# OpenShark v1.6 Roadmap

## Codebase Analysis Summary

### Current State (v1.5)
- 44K lines of Rust, 417 tests passing
- 30+ modules covering agent loop, TUI, LLM providers, tools, session management
- **Integrations**: Hermes, Claw, OpenCode, Claude (delegation bridges)
- **LSP Client**: Sync-only (blocking stdin/stdout), supports definition, hover, document symbols
- **Doctor**: 7 health checks (config, providers, memory DB, cache, skills, binary, sessions)
- **Self-Improve**: Full analysis pipeline — model performance, tool success, trends, prompt effectiveness, session quality scoring
- **Capabilities**: 24 native tools across 9 categories (web, media, memory, productivity, communication, smart_home, platform, agentic, execution)
- **Evolution Engine**: Adaptive state with tool confidence thresholds, model bias tracking, session-triggered analysis
- **Skills**: YAML frontmatter + markdown, trigger-based loading, 5 built-in skills (rust, docker, git, testing, debugging)
- **Sandbox**: Basic tmpdir-based code execution (Rust, Python, Bash, JS), no real isolation
- **JSON Output**: NDJSON event stream (5 event types: ToolCall, ToolResult, Message, Error, Complete)
- **Headless**: `--yolo`/`--json`/`--timeout`/`--max-turns` flags, basic loop with no tool execution
- **Linting**: 6 linter integrations (clippy, eslint, npm, ruff, gofmt, make), parse-and-display only
- **MCP Client**: Full client with stdio + SSE transport, multi-server manager
- **MCP Server**: stdio JSON-RPC server exposing 6 tools (terminal, read_file, write_file, search_files, git, repo_map)
- **Plugins**: Script-based hooks from `.openshark/hooks/` (shell/python)
- **Swarm**: Multi-agent with roles, consensus, persona filtering, agent runner
- **Guardian**: LLM-driven code review with structured findings
- **Watch**: Polling-based file watcher (test/lint/build/custom)

### Key Gaps Identified
1. **Headless mode is incomplete** — no tool execution loop, just sends messages back and forth
2. **LSP is sync-only** — blocks the async runtime, no diagnostics support
3. **Sandbox has zero isolation** — just /tmp files with `timeout`, no namespace/chroot/cgroups
4. **No test runner integration** — no `cargo test`/`pytest` execution or result parsing
5. **Linting is fire-and-forget** — no auto-fix, no inline display in TUI, no feedback loop
6. **Headless/Web API missing** — no HTTP server for external tool integration (MCP server is stdio-only)
7. **Skills are static** — no auto-generation from session patterns despite evolution system
8. **Watch uses polling** — no inotify/fanotify, 500ms sleep loop
9. **JSON output is write-only** — no stdin command interface for programmatic control
10. **No undo/rollback** — no git-based checkpoint/restore after tool execution failures

## Proposed v1.6 Features (8 features)

### 1. Headless Agent Loop Completion
**Description**: Make headless mode actually execute tools. Currently `run_headless()` just loops messages without invoking any tools — it sends a ChatRequest with the user prompt, gets a response, and loops. It needs to parse tool calls from model responses, execute them via the tool registry, and feed results back.

**Why it matters**: Headless/CI mode is the primary interface for automation. Without tool execution, `openshark --yolo "fix the tests"` does nothing useful. This is the #1 functional gap.

**Complexity**: M

**Builds on**: `headless.rs`, `tools/mod.rs` (Tool trait), `json_output.rs`, `agent/mod.rs` (existing agent loop logic)

---

### 2. LSP Async Rewrite + Diagnostics Pipeline
**Description**: Rewrite the LSP client to use async I/O (tokio process + async stdin/stdout). Add `textDocument/publishDiagnostics` handling so the agent can surface real-time type errors, warnings, and code actions. Add `textDocument/references` for find-all-references. Wire diagnostics into the TUI as an inline gutter/status bar and into the agent's edit feedback loop.

**Why it matters**: The current sync LSP client blocks the runtime. Diagnostics are the most valuable LSP feature for a coding agent — they let the agent detect and fix type errors immediately after edits. Without them, the agent is flying blind on correctness.

**Complexity**: L

**Builds on**: `lsp/mod.rs`, `tui/`, `tools/edit.rs`, `linting.rs`

---

### 3. Git Checkpoint & Rollback System
**Description**: Before every agent tool execution that modifies files (edit, write, terminal), auto-create a git stash or commit checkpoint. Add `/undo` slash command to roll back to the last checkpoint. Add `/diff` to show what changed since last checkpoint. Expose checkpoint status in TUI header.

**Why it matters**: Users need a safety net. If the agent makes a bad edit, there's currently no way to undo without manual git. This is table-stakes for every coding agent (Claude Code has this, Cursor has this). The `tools/checkpoint.rs` file already exists but is stubbed.

**Complexity**: M

**Builds on**: `tools/checkpoint.rs`, `tools/git.rs`, `tools/edit.rs`, `slash_commands/`, `session.rs`

---

### 4. Test Runner Integration
**Description**: Add a `TestRunner` module that detects the project's test framework (cargo test, pytest, jest, go test), executes tests, parses results into a structured `TestResult` type (pass/fail/error with file:line:message), and surfaces failures to the agent for auto-fixing. Add `/test` slash command and auto-run tests after code edits when guardian/self-improve detect test files changed.

**Why it matters**: The agent can write code but can't verify it works. The sandbox runs code snippets, but there's no integration with actual project test suites. This creates a tight edit→test→fix loop that dramatically improves code generation quality.

**Complexity**: M

**Builds on**: `sandbox.rs`, `linting.rs` (detection patterns), `tools/`, `guardian/`, `slash_commands/`, `doctor.rs` (add test runner detection)

---

### 5. Lint-Driven Auto-Fix Loop
**Description**: After every file edit, auto-run the detected linter. If the linter reports errors on the edited file, feed the lint errors back to the agent as immediate feedback so it can fix them in the next turn. Add `--fix` support (run clippy --fix, ruff --fix, eslint --fix) as an agent tool. Show lint status in TUI.

**Why it matters**: Currently linting is a manual `/lint` command with no feedback to the agent. A coding agent that can't self-correct lint errors is crippled. This closes the loop: edit → lint → auto-fix → verify.

**Complexity**: S

**Builds on**: `linting.rs`, `tools/edit.rs`, `agent/mod.rs`, `tui/`

---

### 6. HTTP/WebSocket API Server
**Description**: Add an HTTP + WebSocket server mode (`openshark serve --port 8080`) using the existing `axum` dependency (optional `web-api` feature). Expose endpoints: `POST /chat` (send message), `GET /events` (SSE stream of agent events), `GET /status` (session state), `POST /tool/call` (direct tool invocation). Use the JSON output event types already defined.

**Why it matters**: The MCP server is stdio-only. An HTTP API enables browser extensions, IDE plugins, CI pipelines, and remote access. The `web-api` feature flag and `axum` dependency already exist in Cargo.toml but are unused. This is the highest-value unlocked infrastructure.

**Complexity**: L

**Builds on**: `json_output.rs`, `headless.rs`, `mcp_server.rs`, `gateway/`, Cargo.toml (`axum` dep, `web-api` feature)

---

### 7. Smart Skill Auto-Generation
**Description**: Extend the evolution engine to automatically create skills from repeated patterns. When `self_improve` detects a tool-use pattern occurring 3+ times (e.g., "user asks about docker → agent always runs these 5 steps"), auto-generate a skill YAML+markdown file in `~/.config/openshark/skills/auto/`. Add `/skills auto` command to review/accept/reject generated skills.

**Why it matters**: The evolution system tracks patterns but doesn't close the loop — it never creates skills from its own insights. The infrastructure is all there (self_improve, evolution, skills), but the connection is missing. This makes the agent genuinely self-improving rather than just self-analyzing.

**Complexity**: M

**Builds on**: `self_improve/mod.rs`, `evolution/mod.rs`, `skills/mod.rs`, `memory/store.rs`

---

### 8. Real Sandbox Isolation with Resource Limits
**Description**: Replace the current tmpdir-based sandbox with proper isolation using Linux namespaces (via `clone()` with `CLONE_NEWNS`/`CLONE_NEWNET`) or container-based execution (detect docker/podman, run in ephemeral container). Add memory limits, CPU time limits, network isolation, and filesystem read-only mounts. Fall back to current approach when containers aren't available.

**Why it matters**: The current sandbox writes to `/tmp` and runs arbitrary code with the user's full permissions. For a tool that executes model-generated code, this is a security liability. The `security/` module exists but the sandbox doesn't use it.

**Complexity**: L

**Builds on**: `sandbox.rs`, `security/`, `capabilities/execution.rs`, `doctor.rs` (add container runtime check)

## Priority Order

| Priority | Feature | Complexity | Impact |
|----------|---------|-----------|--------|
| 1 | Headless Agent Loop Completion | M | Critical — headless mode is broken without this |
| 2 | Git Checkpoint & Rollback | M | High — safety net, user trust |
| 3 | Test Runner Integration | M | High — edit-test-fix loop |
| 4 | Lint-Driven Auto-Fix Loop | S | High — low effort, big quality improvement |
| 5 | LSP Async + Diagnostics | L | High — but large scope |
| 6 | HTTP/WebSocket API | L | Medium — enables ecosystem |
| 7 | Smart Skill Auto-Generation | M | Medium — differentiation |
| 8 | Real Sandbox Isolation | L | Medium — security improvement |

### Suggested v1.6 Scope
Ship features 1-4 as the core release. Features 5-6 as stretch goals. Features 7-8 can start in v1.6 and land in v1.7.

### Estimated Effort
- Features 1-4: ~8-10 weeks (2 M + 1 M + 1 M + 1 S = ~5 feature-months)
- Features 5-6: ~6-8 weeks additional
- Features 7-8: ~6-8 weeks additional
